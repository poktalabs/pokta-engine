## Decision (CEO review)

**B — Multi-tenant MVP out of the gate.** Ship **both** Mi Pase and Vino Design Build on one
multi-tenant godin-engine + the multi-tenant SPA (already built). Vino is a committed near-term
delivery (signed Phase-1 proposal, 2026-05-28), so the multi-tenant investment is paid-for, not a
forward bet. The marketplace north star (x402, ERC-8004, public authorship) stays deferred.

**Why B over single-tenant:** the deciding facts are (1) Vino is real and near, and (2) **both
tenants' workflows already exist engine-native** — Mi Pase pricing (M1) and Vino
`call-intake → proposal-step → send-step` (the existing `/demo` POC: transcript → CRM → proposal →
email, with two approval gates). So "multi-tenant" is mostly *scoping what exists per tenant + rails*,
not building two product runtimes. Single-tenant would have stranded the Vino POC and forced a later
rewrite of exactly these rails.

**Mode:** HOLD SCOPE — the architecture is fixed at B; this plan makes it bulletproof.

## What already exists (do not rebuild)

- **Tenant key threaded everywhere:** `consumer_id` on `engine_runs`, `engine_quota_ledger`,
  `engine_workflow_state`; required at dispatch (`index.ts:60`), carried to child runs (`index.ts:174`).
- **Mi Pase workflows:** `pricing-draft → pricing-apply-{confident,flagged}` (M1, native, tested).
- **Vino workflows:** `call-intake → proposal-step → send-step` (native, in the registry, the `/demo`).
- **Integration pattern:** `packages/{notion,resend,shopify,mercadolibre}` fail-soft; per-tenant
  resolver keyed by consumer (`worker/src/provider-config.ts`, env-prefix map).
- **The multi-tenant SPA** (this branch): shell, both approval renderers, `/:tenant` routing,
  tenant config, i18n/currency seam. `auth-model.md` already specifies Privy-JWT → consumer.

## The rails B requires (grounded in current gaps)

### Tier 1 — Security rails (REQUIRED before ANY client exposure, tenancy aside)
1. **Bind auth → consumer.** `auth.ts:20` parses `SERVICE_KEYS="consumer:key"` but discards the
   consumer; `index.ts:59` trusts `body.consumer_id`. Fix: map key→consumerId, put it in context,
   and reject/override a mismatched `consumer_id`. Add the **Privy-JWT** path (verify JWT → user →
   consumer) for the SPA (machine service-keys stay for server-to-server). Two auth modes, one
   resolved `consumer` in context.
2. **Scope every read + approval action by the authed consumer.** `GET /v1/runs` (`:117`, consumer
   is optional → leaks all), `GET /v1/runs/:id` (`:109`, no check), `GET /v1/approvals` (`:134`, no
   filter), `approve`/`reject` (`:151,213`, no tenant guard). Filter/guard all by ctx consumer; a
   cross-tenant id → 404, not 403 (don't confirm existence).
3. **Lock the public operator surfaces.** `/demo`, `/dashboard`, `/console` mount before `/v1` auth
   (`index.ts:34-40`) and aggregate all tenants. Gate behind operator auth (or stop serving to
   clients); never expose cross-tenant rollups to a tenant.

### Tier 2 — Tenancy runtime (the actual multi-tenant build)
4. **`engine_tenants` registry.** New table: `tenant_id` (== consumer_id), name, status, currency,
   locale, branding, `allowed_workflows[]`, `secret_prefix`, created_at. Seed `mi-pase` + `vino`.
   Onboarding = insert a row + set secrets. This replaces the hardcoded `ENV_PREFIX`/`TENANTS` maps
   (engine `provider-config.ts` + SPA `TenantProvider.tsx`) with the DB as source of truth.
5. **Per-tenant workflow scoping.** `getWorkflow`/`listManifests` become consumer-aware: a consumer
   can only POST/list workflows in its tenant's `allowed_workflows`. Mi Pase → pricing-*; Vino →
   call-intake/proposal-step/send-step. A direct cross-tenant workflow POST → SKILL_NOT_FOUND.
6. **Per-tenant secret refs.** Keep env-backed for the MVP but key it off the tenant record's
   `secret_prefix` (MIPASE_*, VINO_*), not a hardcoded map — so the encrypted secret store drops in
   later without touching workflows. (Encryption hardening = deferred.)

### Tier 3 — Vino real integrations (the only net-new package work)
7. Vino's workflows run today on demo integrations (Notion CRM, Resend email). Vino's real stack is
   **GoHighLevel** (CRM), **JobTread** (estimates/proposals), **Gmail** (email). New fail-soft
   packages mirroring `packages/notion`/`resend`: `packages/gohighlevel`, `packages/jobtread`,
   `packages/gmail`. The workflow orchestration + gates are unchanged; only the IO adapters are new.
   (Calendar/Twilio/SmartSuite = later.) **This is the largest single chunk of B.**

## Dependency shape (parallel-aware)

```
Tier 1 security rails ──┬─ (1 auth-bind+Privy) ── gates ── (2 read-scope) ── gates ── SPA real wiring
                        └─ (3 lock surfaces)  ── parallel
Tier 2 ── (4 engine_tenants) ── gates ── (5 per-tenant workflow scoping) ‖ (6 secret refs)
Tier 3 ── (7 GHL ‖ JobTread ‖ Gmail packages) ── parallel, independent, gate Vino's real run
SPA ── Privy provider + JWT→apiFetch (P6 from M2 plan) ── gated on (1)
```
Critical path: **auth-bind (1) → read-scope (2) → SPA wiring**. Tier 3 packages parallelize fully and
only gate Vino's *real* (non-demo) execution, so Mi Pase can go live first while Vino integrations land.

## Sequence (recommended)

1. **Tier 1 first** (security) — it's required regardless and unblocks safe Mi Pase exposure.
2. **engine_tenants + per-tenant scoping** (Tier 2) — makes two tenants real and safe.
3. **Wire the SPA** to live `/v1` with Privy (M2 plan's deferred P5/P6, now in scope).
4. **Vino integrations** (Tier 3) in parallel — Mi Pase ships on Shopify/ML while GHL/JobTread/Gmail land.
5. Mi Pase to production prices stays gated on margin validation (unchanged, M1 T12).

## Deferred (NOT in this MVP)
- Encrypted/managed per-tenant secret store (env-backed keyed by tenant record for now).
- x402 metering, ERC-8004 reputation, public workflow authorship (the marketplace).
- Self-serve tenant onboarding UI (insert-a-row + set-secrets is operator-run for 2 tenants).
- Calendar/Twilio/SmartSuite (Vino), LG/Mabe/Slack/Sheets (Mi Pase).
- B2 "port Vino's external agent runtime" — N/A; the workflows are already engine-native.

## Risks / open items
- **Two auth modes** (service-key + Privy-JWT) resolving to one `consumer` in context — get this seam
  clean; it's the security spine. (`auth-model.md` is the start.)
- **Vino integration credentials** (GHL/JobTread/Gmail API access) — a real-world dependency on Carlos,
  like Mi Pase's Shopify/ML creds. Name the owner + timeline.
- **`decided_by` is unauthenticated** (`index.ts:154`) — with Privy, bind the approver identity for audit.
- The hardcoded tenant maps exist in **two places** (engine `provider-config.ts` + SPA
  `TenantProvider.tsx`); the registry must become the single source both read (engine via DB, SPA via
  an authed `/v1/tenants/me` endpoint) to avoid drift.

## Eng Review (2026-06-08) — resolved decisions

- **Slicing: 3 sequential PRs** — PR1 Tier-1 security (ships + deploys alone, unblocks safe Mi Pase),
  PR2 Tier-2 tenancy runtime, PR3 Tier-3 Vino integrations. Each independently testable + reversible
  (incremental-over-revolutionary; low blast radius on the live M1 auth path).
- **Read/approval isolation = ONE app-level scoped-query helper** `forConsumer(db, consumerId)`. Every
  `engine_runs`/`engine_approvals`/`engine_workflow_state` read+write goes through it; routes cannot
  query unscoped. Single chokepoint + single audit point (DRY). Postgres RLS = deferred hardening (TODO).
- **Auth = ONE Hono middleware** resolving EITHER service-key (machine) OR Privy-JWT (browser) to a
  single `c.set('consumer', {id, identity, mode})`. Routes read context, NEVER `body.consumer_id` for
  identity. Privy verified via **JWKS** (don't hand-roll). `decided_by` binds to `ctx.identity`, not a
  body string (`index.ts:154`).
- **Cross-tenant access → 404, not 403** (don't confirm existence of another tenant's resources).
- **`engine_tenants` is the single source of truth** — replaces BOTH hardcoded maps
  (`provider-config.ts` `ENV_PREFIX` + SPA `TenantProvider.TENANTS`). SPA reads an authed
  **`GET /v1/tenants/me`**.

## Test matrix (required — security; no AUQ per regression rule)

```
AUTH            no key/JWT → 401 · valid service-key → consumer resolved
                Privy JWT valid → consumer · expired/wrong-aud/bad-sig → 401 (fail CLOSED)
ISOLATION       tenant A cannot: GET B's /runs/:id (404) · see B in /runs · GET/approve/reject B's
                approval (404) · POST a workflow not in A's allow-list (SKILL_NOT_FOUND)
SCOPED HELPER   unit: every accessor injects WHERE consumer_id; a raw unscoped read is unreachable
WORKFLOW SCOPE  listManifests(consumer) returns only allowed; getWorkflow cross-tenant → not found
VINO PKGS (T3)  fail-soft per package (ok / 4xx / unconfigured), mirroring shopify/mercadolibre tests
REGRESSION ★    existing M1 single-tenant mi-pase chained flow still green post-scoping (CRITICAL)
```

## Failure modes (each new codepath)

- **Privy JWKS unreachable** → 401 fail-CLOSED (never fail-open). Test + clear error. (test ✓ / handled ✓ / visible ✓)
- **Tenant not in registry** at dispatch → reject `TENANT_UNKNOWN`, never default to a tenant. (test ✓ / handled ✓ / visible ✓)
- **Scoped-helper bypass** (a dev writes a raw `db.select` from `engine_runs` in a route) → caught by
  the isolation suite + a CI grep gate forbidding raw engine_* selects outside the helper. (test ✓ / gate ✓)
- **Missing `secret_prefix`/creds for a tenant** → integration fail-soft (existing IntegrationResult). (handled ✓ / visible ✓)
- Critical-gap check: the *current* state (cross-tenant read, no guard, silent) is exactly the gap the
  scoped helper + isolation suite close. No new silent-failure path introduced.
