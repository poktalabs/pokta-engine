## Problem Statement

Build a **client-delivery workspace** (a Vite + React SPA) over godin-engine where a client
self-manages the engine-work we deliver for them. First surface + first client: **Mi Pase**
(operator **Dalia**, champion **Rodrigo Ortiz**) running their **daily pricing pipeline** — schedule
or manually trigger the run, review the flagged price changes, approve, and have approved prices
written to Shopify. Built **single-tenant (Approach A)** for Mi Pase. The validated pipeline is
**local TS/JS not shaped for the engine**, so it is **rewritten** into the engine's `manifest` +
`run(input, ctx)` chained-approval contract — preserving the validated pure logic, re-shaping
orchestration + IO. Not imported/wrapped as-is.

## The product: a client-delivery workspace (Vite SPA), organized around engine concepts

The front-end is a **Vite + React SPA** (same stack as godinez-studio's shipped production code;
PoktaCare is doing similar). It's a **client workspace**, but **everything in it is
workflow/engine-related** — not a generic portal. Surfaces:
- **Workflows** — the client's workflows (e.g. daily pricing): manual **trigger**, **schedule**
  (daily at a set time), run history, live status, and the **approval** queue (review + approve/reject).
- **Integrations** — the client's integrations with **status**: validated (**MercadoLibre**,
  **Shopify**), in-active-development (LG/Mabe portals, Slack, Sheets), each able to carry **reports
  + relevant data** (e.g. the competitor e-commerce **metadata research**).
- **Reports / research** — engine-produced analysis surfaced to the client. Concrete near-term one:
  research how each competitor (Liverpool, Amazon MX, Elektra, Coppel) structures e-commerce
  **metadata**, to propose fixes to Mi Pase's product data (metadata discrepancies currently hurt
  price-match accuracy vs MercadoLibre).
- **(later)** workflow design.

It talks to the engine over `/v1` + a few new endpoints (schedules, integration status, reports).
**Discipline:** build the SPA **shell + the pricing surface** now; integrations/reports/research
surfaces are added as concrete client needs land (the metadata research is the likely 2nd). Shell
extensible; day-one scope = pricing.

## The workflow: Mi Pase daily pricing (grounded in the real code)

Ingest supplier costs (LG/Mabe/Excel/PDF — later) + competitor prices (MercadoLibre MX live via
OAuth now; Coppel/Elektra/Liverpool/Amazon later) → match to the Shopify catalog → run the **frozen
8-branch pricing logic** (15% margin floor, to confirm) → produce recommended prices, **classified
confident vs flagged** → write approved prices to **Shopify** (1% **anti-thrash** guard).

**Engine mapping — chained runs + a flagged-only gate (D-8):**
- `pricing-draft` (runtime `agent`/`sandbox`): pull costs + competitor prices, match, run pricing
  logic → output `{ confident[], flagged[] }`. **No Shopify write.**
- **Both apply paths are live + shown** (the Shopify account is a **developer account**, so no
  production-price risk — we can demonstrate both):
  - **Auto-apply** — `pricing-apply-confident` writes the confident set with **no gate**.
  - **Review-then-apply (flagged batch)** — Dalia reviews the flagged cases as **one batch
    approval** in the dashboard (the per-run version of the "Rodrigo G1 sign-off") →
    `pricing-apply-flagged` writes them.
- **Writes target the Shopify DEV store.** `shopify-write.ts` is already isolated ("NOT called from
  Stage A"). **Margin validation gates the PROD-store cutover (real prices), not the dev demo of
  both flows** — until the 15% floor is validated + G1 signed, we stay on the dev store.
- **Scheduling:** runs **daily at a set time** (`boss.schedule` cron) AND is **manually
  triggerable**; both managed from the Workflows surface.

**Rewrite scope (keep the validated brain, re-shape the shell):**
- **Keep ~as-is (tested, pure):** `pricing-logic.ts`, `matching-score.ts`, `aggregates.ts`,
  `ml-domain-mapping.ts`, brand/model extraction, etc.
- **Re-orchestrate:** `run-pipeline.ts`/`run-live-prices.ts` → the `run()` halves (draft / apply),
  split at the gate, with the confident/flagged classification.
- **Adapt IO:** `shopify-client.ts`, `ml-api-client.ts`/`ml-oauth.ts`, `shopify-write.ts` read keys
  from **per-tenant `ctx`/config** (not `process.env`). `state.ts` anti-thrash → engine/per-tenant state.
- **New integration packages** (mirroring `packages/notion`/`resend`, fail-soft): `packages/shopify`
  + `packages/mercadolibre`.

## North Star (informs seams, rules out C — NOT built now)

Long-term: a multi-tenant **workflow marketplace** (authors publish; consumers run with their own
keys; x402 per-execution payments; ERC-8004 reputation). Two consequences for this build:
1. **Rules out Approach C** (one deploy per client) — a marketplace is **shared**, not N isolated
   deploys. (Mel's point; C rejected.)
2. **Sets cheap forward-compat for A:** `consumer_id` is the future `tenant_id`; the `run` primitive
   is the eventual x402 meter (dispatch txn) + ERC-8004 signal (terminal status). Good to know,
   **nothing to build now.** Multi-tenancy, payments, reputation all deferred until real signal.

## Target User & Narrowest Wedge

- **Wedge (build now):** Dalia, in the Vite workspace, schedules + manually triggers the Mi Pase
  pricing run, reviews flagged cases, approves the batch, prices write to the Shopify **dev store** —
  single-tenant, seeing only Mi Pase data.
- **NOT now:** multi-tenancy, **prod-store cutover** (dev store only until margin validated), the

## Premises (agreed)

1. **Single-tenant (A)** for Mi Pase; multi-tenant marketplace is north star, not this build.
2. Front-end is a **Vite + React SPA** client workspace (godinez-studio stack), organized around
   engine concepts; ship the **shell + pricing surface** first.
3. The pipeline is **rewritten** into the engine contract (chained runs + flagged-only gate),
   preserving the validated pure logic.
4. **Both apply paths shown** (auto-apply confident + review-then-apply flagged) against the
   **Shopify dev store** (developer account → safe); margin validation gates the **prod-store
   cutover**, not the dev demo.
5. **Scheduling (daily) + manual trigger** are dashboard-managed (`boss.schedule`).
6. Day-one integrations = **Shopify + MercadoLibre** only.
7. Build A **forward-compatibly** (consumer_id = future tenant key; config not hardcode).

## Approaches Considered

- **A — single-tenant, Vite SPA workspace + rewritten pipeline (CHOSEN):** deliver Mi Pase now on a
  real client front-end; forward-compatible toward multi-tenant. Effort: human ~2-3 wk / CC ~2-4 days
  (SPA shell + pricing surface + workflow rewrite + 2 integration packages + scheduling).
- **B — multi-tenant engine (DEFERRED, north star):** the marketplace foundation; premature now.
- **C — one deploy per client (REJECTED):** a marketplace is shared; per-deploy fights the north star.

## Recommended Approach: A

**Engine side:**
1. **Rewrite the pipeline** into `workflows/` as a chained pair (`pricing-draft` → flagged gate →
   `pricing-apply-flagged`; plus `pricing-apply-confident` behind the `marginValidated` flag),
   preserving pure logic, IO keys via `ctx`.
2. **`packages/shopify` + `packages/mercadolibre`** (fail-soft integration clients).
3. **Scheduling** — `boss.schedule` daily cron per workflow/tenant + a manual-trigger endpoint.
4. **Scope the read surface** by `consumer_id` (overview rollups recomputed per-tenant); **bind keys
   → consumer** (`auth.ts` TODO); **lock down** public demo/operator surfaces.
5. New endpoints the SPA needs: schedule CRUD, integration status, reports.

**Front-end side:**
6. **Vite + React SPA** (godinez-studio stack) — workspace shell + the **Workflows** surface
   (trigger/schedule/runs/approvals) + a first **Integrations** surface (status of ML + Shopify).
   Auth via the client's `X-Service-Key`.

**Forward-compat (free):** consumer_id = tenant key; client + workflows + keys from config.
**Not now:** multi-tenant machinery, x402/ERC-8004, auto-apply (pre-margin), in-dev integrations,
research/workflow-design surfaces, encrypted secret store.

## Success Criteria

- Dalia logs into the Vite workspace, sees **only Mi Pase** data, **schedules** the daily run +
  can **trigger** it manually.
- A run computes recommended prices and shows **both paths**: **auto-applies confident** ones and
  **opens one batch gate** for flagged cases; **approve** writes them to the Shopify **dev store**,
  **reject** stops cleanly; failures (incl. partial writes) are surfaced. (Prod-store cutover gated
  on margin validation.)
- The rewritten pipeline runs natively under engine lifecycle/governance/observability (a real
  `workflows/<id>/` module, not an external wrap); pure-logic tests carry over green.
- No unauthenticated path can read/trigger/approve Mi Pase data; demo/operator surfaces not served.
- The Integrations surface shows ML + Shopify as validated; the shell can host a 2nd surface
  (research) without a rewrite. Adding multi-tenancy later is additive.

## What I noticed about how you think

- You keep the front-end **disciplined to the engine** — "while it will be used for client-delivery,
  everything will still be workflow-related." That one sentence is what stops a delivery console from
  rotting into a generic CRM. Vision as a boundary, not a license.
- You reach for **proven internal precedent** ("use Vite... we've shipped production code in
  godinez-studio") instead of greenfield. Same instinct as keeping the validated pricing logic:
  reuse what's earned its trust.
- You separate **delivery now** (Mi Pase, single-tenant) from **direction** (multi-tenant
  marketplace) cleanly, and you let the direction *kill wrong options* (C) without letting it
  inflate the build. That's the discipline that ships a first paid client without mortgaging the
  platform.
