# Claude Design prompt — governed agent-workflow workspace (high-fidelity)

Generated by /plan-design-review on 2026-06-08. Paste the block below into Claude Design (which has

**Design intent (read first):** this is **not** a Mi Pase app. It is a tenant-agnostic
**client-delivery workspace** over godin-engine where a business self-manages the governed agent
workflows we run for them. The first two tenants are deliberately different so the abstraction is
forced to generalize:

- **Mi Pase** — Mexican high-mix online retailer. Workflow: daily pricing. **One batch gate** over
  many similar flagged price changes. One write target (Shopify). MXN.
- **Vino Design Build** — Napa design-build / remodeling firm (CEO Carlos Barragan). Workflows: lead
  qualification, proposal drafting, email triage. **Every external action gated individually** (a
  drafted email, a CRM pipeline move, a committed estimate) across 7+ systems, with risk tiers. USD.

The universal heart is the **approval queue** with a **pluggable item renderer**: a price-diff batch
table for Mi Pase, a single-action draft-preview card for Vino. The shell, workflows, runs,
integrations, and reports are all generic and **per-tenant themed**.

**Language:** **English-first**, but architect for **i18n** — a locale switcher, all copy from a
string catalog, locale-aware number/date/currency (MXN for Mi Pase, USD for Vino). Spanish (es-MX) is
the first secondary locale (Mi Pase's operators are Mexican; the underlying pipeline already emits
English + Spanish).

Reference, not target: the engine ships an internal `/console` (operator view — Jobs, Policies,
Database/State, all tenants, engine jargon). Reuse its **visual DNA** (left-nav workspace, status
pills, a horizontal pipeline-flow graphic with an amber approval gate, Inter + JetBrains Mono, dark
surface, amber accent) but **not** its information model — the client never sees engine internals.

## ⬇️ COPY FROM HERE

Design a **governed agent-workflow workspace**: a web app where a business operator self-manages the
automated workflows an AI engine runs for them — triggering or scheduling runs, watching results, and
**approving the actions that need a human** before anything is written to an external system. Produce
**high-fidelity, production-ready** screens. **Use the loaded Design System for every token** (color,
type scale, spacing, radius, elevation) and **every component** (buttons, tables, badges, inputs,
cards, nav, tabs, modals). Do not invent new colors or fonts; where a component is missing, compose
it from existing primitives. Desktop-first (operators work at a desk); specify responsive behavior.

This workspace is **multi-tenant and themeable** — the same product is delivered to different clients,
each with their own brand, locale, currency, workflows, and integrations. **Do not hardcode any one
client.** Show the shell themed for **two contrasting tenants** so the abstraction is visible:

- **Tenant A — Mi Pase** (online retailer, Mexico, MXN, Spanish-capable operators)
- **Tenant B — Vino Design Build** (design-build construction, Napa CA, USD, English)

### Core domain model (keep the UI faithful to this)
- **Workflow** — a named automated process. Has a **trigger** (scheduled cron, event-driven, or
  manual), run **history**, live **status**, and a **governance policy** (some results auto-apply
  within policy; some open an **approval gate**).
- **Run** — one execution of a workflow. Produces results split into **auto-applied** (within policy,
  done) and **needs approval** (held at a gate). Can succeed, fail, or partially fail.
- **Approval** — the universal human-in-the-loop gate: a proposed action (or batch of actions) the
  engine will **not** commit until a human approves. Every approval shows **what will change, in which
  system, and its risk level**, with approve / reject and an audit trail (who decided, when).
- **Integration** — an external system the engine reads from or writes to, with a connection
  **status** and a **risk tier** for writes (e.g. external email = high, financial/estimate =
  very-high, calendar = low).
- **Report** — engine-produced analysis surfaced to the client.

### Internationalization (build it in, don't bolt it on)
- English default; a visible **locale switcher** (e.g. EN / ES-MX). All visible strings come from a
  catalog — design assuming text length varies by language (German/Spanish run ~30% longer; don't let
  buttons or table headers break).
- **Locale-aware formatting**: currency per tenant (`$12,499.00 MXN` for Mi Pase, `$48,500.00` USD for
  Vino), thousands separators, dates, and percentages. Currency is a tenant setting, not a constant.

### The visual DNA to honor (from our engine console, elevated to client-grade)
Left-nav workspace shell; compact status **pills** (green = applied/approved/ok, amber =
pending/needs-review, red = failed/rejected, grey = idle/draft); a horizontal **pipeline-flow**
graphic with an amber **approval gate** node; summary **stat tiles**; dense but readable **tables**;
**risk** treated as a first-class visual signal. Make it warm, trustworthy, client-grade — not an
internal ops tool. Trust is earned at the pixel level: every screen makes it obvious what is about to
change, where, at what risk, and how to stop or undo.

### Screens to design (all high-fidelity, all states)

**1. Workspace shell (show themed for BOTH tenants)**
- Left nav, generic sections: **Workflows**, **Approvals** (with a pending-count badge — this is the
  operator's daily landing spot), **Integrations**, **Reports**. A settings/locale control.
- Top bar: the **tenant's** brand lockup (Mi Pase vs Vino — prove the theming), the active section,
  the signed-in user (avatar + name — the user is whoever logged in; don't hardcode a person), and a
  calm **environment badge** when writes target a sandbox (Mi Pase runs on a Shopify **test store**).
- Hierarchy: nav (where am I) → page title + primary action → content. Trunk test: with content hidden
  you still know whose workspace this is and which section you're in.

**2. Workflows — list + detail**
- **List**: each workflow as a row/card with its trigger (e.g. "Daily · 6:00 AM" or "On new lead" or
  "Manual"), last-run outcome pill, and pending-approval count. Mi Pase shows "Daily Pricing"; Vino
  shows "Lead Qualification", "Proposal Drafting", "Email Triage".
- **Detail (worked example: Mi Pase "Daily Pricing")**: first thing the eye lands on is **today's run
  status** + the **primary action** (▶ "Run now"); a **schedule** control ("Runs daily at 6:00 AM",
  editable) that also supports **event-driven** and **manual-only** modes (for Vino's workflows); the
  **pipeline-flow** graphic showing the live stage; **run history** newest-first (date, # auto-applied,
  # approved, # rejected, outcome pill).
- States: **empty / never-run** (warm first-run state explaining what a run does + one ▶ CTA, never
  "No data"); **running** (live progress, current stage); **failed** (plain-language cause +
  "Retry"); **idle / up-to-date** (last summary + next scheduled time).

**3. Run detail**
- **Summary tiles**, scannable in 3s. Mi Pase example (real shape): **Products analyzed 316**,
  **auto-applied 248**, **needs review 41**, **no change 27**.
- The **auto-applied** set reads as *done* (collapsed: "248 prices applied ✓", expandable to a table) —
  information, not a competing task.
- The **needs-review** set is the call to action → routes into Approvals (screen 4).
- States: **all clear** (zero needs-review — celebratory success: "Everything applied automatically,
  nothing to review", warm, not an empty table); **partial failure** surfaced honestly (see screen 4).

**4. ★ Approvals — the hero, with a PLUGGABLE item renderer (design BOTH variants)**
This is where the operator lives and where trust is won or lost. One generic frame — a queue of
pending approvals, each showing **what changes, where, risk, and reason**, with **Approve** / **Reject**
and audit trail — but the **item body adapts to the workflow**. Design two item renderers to prove the
abstraction generalizes:

  **4a. Batch renderer — Mi Pase flagged price review (the high-volume case).**
  A **review table**, one row per flagged product, approved as **one batch** (mirrors the real
  one-shot sign-off). Columns:
  - Product (name + thumbnail; **gracefully handle a long name** like "Asiento de bidé eléctrico Nuur
    con control remoto" — truncate + tooltip, never break layout) + SKU (muted, secondary) +
    category chip (the catalog is **high-mix**: phones, audio, computing, washers, **motorcycles**,
    **mattresses**, **fragrances**, microwaves, water heaters, AC — vary the mock rows accordingly).
  - **Current price** (MXN).
  - **Suggested price** (MXN) — visually primary, with a **Δ delta** chip (▲/▼ %, color-coded; a large
    swing reads as a gentle warning, not alarm).
  - **Competitor reference** — the price that drove it **with its source label** (e.g. "Mercado Libre
    MX", and quietly note when a source is a live vs estimated feed); a subtle "view ↗".
  - **Resulting margin** (%), with a clear treatment at/below the **15% floor**.
  - **Why flagged** — the real reasons are: **"cost unknown — can't verify margin"** or **"price
    already below the margin floor (anomaly)"**. (Do NOT invent "thin margin / big swing" reasons —
    those auto-apply; only genuine review cases land here.)
  - Batch actions: **"Approve all & apply"** (primary) and **"Reject"** (calm destructive). Allow
    excluding individual rows before approving. Confirm states plainly: **"This will update N prices in
    Shopify (test store)"** with the count and a back-out.
  - Scale: design for a long table (316-scale) — sticky header, sticky action bar, comfortable
    scanning, unambiguous count; if paginated, the batch scope stays obvious.

  **4b. Single-action renderer — Vino Design Build (the high-stakes, per-action case).**
  Here each approval is **one discrete action** the agent drafted but will not commit without Carlos.
  Design a focused **action card / detail panel** for examples like:
  - "Send follow-up email to lead **Maria G.**" → show the **drafted message** (preview), the
    recipient, the channel (Gmail), and **risk: High (external send)**.
  - "Move lead to **Qualified** in GoHighLevel" → before/after pipeline stage; **risk: Medium (CRM
    write)**.
  - "Commit estimate **#1042** to JobTread" → line-item summary, total **$48,500.00**; **risk:
    Very high (financial/proposal)**.
  Each shows **what / where / risk / why**, the drafted content to inspect, and **Approve** / **Reject**
  (with optional "edit before approving" for drafts). Risk tier is a prominent, color-and-label signal
  (never color alone).

  - States for BOTH renderers: **default** (items pending); **empty / inbox-zero** (warm "You're all
    caught up", not a blank); **submitting** (applying — disable double-submit, progress vs count);
    **success** ("N applied" / "Email sent" + path back); **partial failure** (e.g. 38 of 41 applied,
    3 failed — show exactly which, the reason, and **"Retry failed"**; never hide a partial write,
    never claim full success); **rejected** (clean stop, nothing written, clear confirmation, reason
    captured for audit).

**5. Integrations**
- Cards per integration with **connection status** and **write risk tier**. Show both tenants' sets:
  - Mi Pase: **Shopify** (connected, test store), **Mercado Libre MX** (connected — competitor feed);
    **Coppel / Elektra / Liverpool / Amazon MX** as additional competitor sources (some estimated /
    not-yet-live — show calmly, "estimated feed", not broken).
  - Vino: **GoHighLevel** (CRM), **JobTread** (estimates — very-high risk), **Gmail**, **Google
    Calendar**, **Twilio**, **SmartSuite** — varied statuses and risk tiers.
- Each card can host a small **report/data** slot (forward-looking — e.g. competitor-metadata research
  for Mi Pase). It's a **status surface**, not a generic 3-up feature grid.

**6. Reports**
- A simple index of engine-produced reports, themed per tenant. Mi Pase: "Daily pricing impact"
  (e.g. a real run: 35 analyzed, 32 lowered to competitor, avg −4.9%, **total −$56,260.04 MXN**),
  "Competitor metadata research". Vino: "CEO execution brief", "Lead pipeline health", "Stale leads".
- One report opened to show the layout (summary + a table/chart) — don't over-build; prove the slot.

### Cross-cutting requirements (every screen)
- **States are features.** Every screen ships empty, loading, error, and success — designed. Empty
  states have warmth + a primary action, never "No data found".
- **Hierarchy on purpose.** Name first / second / third. The primary action is unmistakable;
  done/auto-applied info never competes with the review task; **Approvals** with a pending count is the
  operator's home base.
- **Risk is a first-class signal.** Approval items and integration writes carry a visible risk tier
  (low → very-high) by icon + label + color, never color alone.
- **Trust cues.** Always show what will change, where, the count, and the risk; always offer a back-out
  before a write; surface partial failures honestly; show the sandbox/test-store badge when applicable.
- **Anti-slop.** No generic SaaS hero, no decorative 3-column feature row, no cards where a table or a
  single strong element is clearer. The approval surface is the product, not chrome around it.
- **i18n + locale.** English-first with a locale switcher; strings from a catalog (allow ~30% longer
  translations without breakage); per-tenant currency/number/date formatting.
- **Edge cases to render (don't hide them):** a 40+ char product/lead name; a price at the 15% floor; a
  large (>30%) swing; a partial-write failure; a zero-needs-review success day; a very-high-risk Vino
  estimate approval; inbox-zero approvals.
- **Accessibility.** Sufficient contrast; meaning never by color alone (pair icon/label); 44px touch
  targets; visible focus; keyboard-navigable tables, queue, and approve/reject actions.
- **Responsive.** Desktop-first; specify how the shell nav collapses and how the wide review table /
  action panel reflows on tablet/narrow viewports (stay usable, don't just stack).

### Deliverable
High-fidelity screens for all six, each screen's key states shown, the **shell themed for both Mi Pase
and Vino**, and **screen 4 (Approvals) designed in both renderer variants (4a batch, 4b single-action)**
as the strongest, most resolved work. Use the loaded Design System throughout.

## Why these choices (design-review rationale)

- **Tenant-agnostic by construction.** Two deliberately opposite tenants (Mi Pase batch/low-risk/daily
  vs Vino per-action/high-risk/event-driven) force the UI into a real abstraction instead of an
  ecommerce-pricing skin. The shell, workflows, runs, integrations, reports are generic; only the
  approval **item renderer** and run-summary are domain-specific.
- **Approvals is the universal heart.** Both clients' core value is human-in-the-loop governance —
  nothing writes externally without sign-off. That's why Approvals is its own nav home with a pending
  badge, and why screen 4 gets two renderer variants and the deepest state coverage (partial-write
  failure is the state most prompts forget, and the one that most erodes trust on real writes).
- **Risk tiers are cross-tenant.** Mi Pase's "this writes a price to Shopify" and Vino's "this sends an
  email / commits a $48k estimate" are the same trust problem at different stakes — a shared risk
  signal serves both.
- **Grounded data, not invented.** Mi Pase is high-mix (phones, audio, motorcycles, mattresses,
  fragrances, appliances), MXN $2,199–$479,900, real decision logic (flag = cost-unknown or
  below-floor-anomaly only), real run numbers (316 SKUs; a 35-item run cut $56,260.04 MXN). Vino spans
  GHL/JobTread/Gmail/Calendar/Twilio with risk tiers. Use these so the demo rings true.
- **English-first + i18n** because the product serves both a US (Vino) and Mexican (Mi Pase) client;
  locale/currency are tenant settings, and Spanish is a first-class secondary locale, not a fork.
- **DNA reuse, model rejection.** The `/console` look (left-nav, pills, amber-gate flow, dark surface)
  carries over; its information model (all-tenants, DB tables, engine jargon) does not — that's what
  keeps this client-grade.
